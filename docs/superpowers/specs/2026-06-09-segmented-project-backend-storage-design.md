# Segmented TTS — Backend Project Storage Design

**Date:** 2026-06-09
**Status:** Approved (pending user spec review)
**Author:** Brainstorming session

## 1. Background and Motivation

The segmented TTS editor introduced in `2026-06-06-segmented-tts-editor-design.md` ships as a **frontend-only** feature. Projects, chapters, segments, parameter locks, and audio references all live in browser IndexedDB. The backend (`backend/app/models/segmented_project.py`) contains a **dormant skeleton** that:

- is not imported in `models/__init__.py`, so `Base.metadata.create_all()` never creates the tables
- encodes a v1 flat `segments` shape that no longer matches the frontend (frontend is now `schema_version=2` with `chapters[]`)
- has no companion service / API / routes

A `source='segmented_tts'` column exists on `tts_results` but is never written. The `system_configs` storage-mode toggle covers only `tts` and `speech_to_text` history; it does not reach the segmented editor.

This spec closes that gap: **add complete backend project storage for the segmented editor**, with the user's selected model

> Database as authoritative source + dedicated asset directory `uploads/segmented/{project_id}/` for text, SSML, and audio (mp3).

## 2. Goals and Non-Goals

### Goals

- `storageMode = backend` makes the segmented editor's project data persisted exclusively on the backend (DB + filesystem). IndexedDB is **not** the authority in backend mode.
- Frontend retains IndexedDB as a **draft / failure-protection cache** only; backend is always the canonical version.
- Provide a one-time **migration flow** so existing IndexedDB-only projects can be moved to the backend.
- Match the frontend v2 schema (`schema_version=2`, `chapters[]`).
- All segmented audio is stored as **mp3** under the project directory.
- Stale detection (segment text / params diverged from `generated_params`) and per-segment regeneration both work in backend mode.

### Non-Goals

- Multi-user permissioning
- Real-time collaboration / WebSocket sync
- Backend-side audio concatenation and SRT generation (frontend keeps doing these for v1)
- Cloud sync / conflict resolution across devices beyond a simple `base_updated_at` check
- Backend export endpoints (`/export`) — the endpoint may exist as a placeholder but the implementation is out of scope for v1
- A per-segment "PATCH" API — frontend uses `PUT` to save the full project state
- Incremental PATCH optimization for very large projects

## 3. Architecture

### Storage layers

backend mode splits persistence into two layers:

1. **Database is the authoritative source.** Holds project / chapter / segment rows plus all metadata and the `updated_at` timestamps used for sorting and conflict detection.
2. **Filesystem is the project asset mirror.** Each project has a dedicated directory containing original text, per-chapter text, per-segment text and SSML snapshots, segment audio (mp3), and a `manifest.json` snapshot.

### Frontend boundary

`TTSSynthesis.tsx` and `useSegmentedProject` stop depending directly on `segmentedProjectDB.ts`. They depend on a unified storage interface:

```ts
segmentedProjectStorage
  ├─ indexedDB adapter   // frontend mode
  └─ backend adapter     // backend mode
```

The unified interface exposes:

- `listProjects()`
- `getProject(id)`
- `saveProject(project, options)`
- `deleteProject(id)`
- `migrateLocalProjectsToBackend()`

The page / hook decides which adapter to instantiate based on `useStorageMode`.

### Backend boundary

```text
backend/app/api/segmented_projects.py           # FastAPI routes
backend/app/services/segmented_project_service.py   # business logic, DB ↔ filesystem reconciliation
backend/app/models/segmented_project.py        # SQLAlchemy models (replaces skeleton)
```

The existing skeleton at `app/models/segmented_project.py` is replaced; `models/__init__.py` is updated to register the new models so `Base.metadata.create_all()` picks them up at startup.

### TTS parameter fields in detail

Each segment stores three TTS-parameter-related fields with distinct semantics:

- `params` — the user-set per-segment overrides that apply right now
- `locked_params` — array of parameter keys the user has explicitly locked; global / chapter default changes do not affect these
- `generated_params` — full parameter snapshot used to produce the current `current_audio_path`; used for stale detection

Effective parameters for a segment are computed as:

```text
effective = chapter.default_params ⊕ segment.params   (segment overrides win)
```

Stale detection compares `effective` against `segment.generated_params`.

### Audio reference design

In backend mode, segmented audio does **not** use the global `tts_results` table. Each segment points to a relative path under its own project directory:

```text
uploads/segmented/{project_id}/chapters/{chapter_id}/segments/{segment_id}.mp3
```

The `current_audio_path` column stores the relative path from the project root, e.g. `chapters/{chapter_id}/segments/{segment_id}.mp3`. This isolates the segmented editor's audio lifecycle from the global TTS history table.

### Existing TTS services

The TTS engines (Edge, CosyVoice, MiMo) currently output wav / pcm. The backend transcodes to **mp3** before writing to the project directory. Encoding uses `ffmpeg` invoked via Python subprocess; runtime existence is checked at startup with a clear log warning if missing, in which case the project falls back to writing wav and flagging the row for manual conversion.

### Model registration

`backend/app/models/__init__.py` is updated to import the new models so that `Base.metadata.create_all()` creates the tables on startup. The old skeleton module is replaced, not imported alongside.

## 5. API Design

APIs are **business-capability-oriented**, not action-mirrored. Most editor actions (text edits, split, merge, delete, reorder) are still produced by the frontend reducer; the resulting project state is then saved via `PUT`. Audio generation is the only operation that requires a dedicated endpoint because it produces files and updates generated fields.

### 5.1 Project CRUD

```http
GET    /api/segmented-projects
POST   /api/segmented-projects
GET    /api/segmented-projects/{project_id}
PUT    /api/segmented-projects/{project_id}
DELETE /api/segmented-projects/{project_id}
```

**`GET list`** returns lightweight rows (no `original_text`, no `chapters` / `segments`).

**`GET detail`** returns the full project including chapters and segments, in the same shape the frontend uses.

**`POST`** accepts the full project object; the backend assigns nothing — `id` is supplied by the caller.

**`PUT`** is a full-state save. The service layer reconciles chapters and segments:

- rows missing in the request body are deleted (and their audio files cleaned up)
- rows present are inserted or updated
- the request body's chapter / segment `id` is treated as the source of truth for identity

**`DELETE`** removes the DB rows and the project directory. Directory removal failure is non-fatal and logged for background cleanup.

### 5.2 Segment audio

```http
POST /api/segmented-projects/{project_id}/chapters/{chapter_id}/segments/{segment_id}/synthesize
GET  /api/segmented-projects/{project_id}/audio/{chapter_id}/{segment_id}
```

`synthesize` request body:

```json
{
  "params": { "engine": "edge_tts", "voice_id": "...", "speed": 1.0 },
  "text": "optional override; defaults to segment.text",
  "ssml": "optional override; defaults to segment.ssml",
  "keep_previous": true
}
```

Service flow:

1. Verify project / chapter / segment relationships
2. Merge `chapter.default_params` + `segment.params` + request `params` (request wins)
3. Call the engine (Edge / CosyVoice / MiMo)
4. Move existing `current_audio_path` to `previous_audio_path` (delete after; keep if `keep_previous=false`)
5. Transcode to mp3
6. Write `chapters/{chapter_id}/segments/{segment_id}.mp3`
7. Update `current_audio_path`, `previous_audio_path`, `duration_sec`, `generated_params`, `generated_at`
8. Touch `project.updated_at`
9. Return the updated segment (or full project)

`GET audio` returns the mp3 with `Content-Type: audio/mpeg`.

### 5.3 Split / resegment

```http
POST /api/segmented-projects/{project_id}/chapters/{chapter_id}/split
```

Request:

```json
{
  "text": "raw text to split",
  "mode": "rule",
  "delimiters": ["，", "。", "！", "？"],
  "replace_strategy": "preview_only"
}
```

`replace_strategy` options:

- `preview_only` — return segments, do not write DB
- `replace_chapter_segments` — write DB, return updated project
- `append_to_chapter` — write DB, append after last segment
- `insert_after_segment` — requires `after_segment_id`

For v1, `preview_only` and `replace_chapter_segments` are required; the others are optional.

### 5.4 Migration

```http
POST /api/segmented-projects/migrate
```

Request:

```json
{
  "projects": [ { "...": "full project" } ],
  "audios": [
    { "project_id": "uuid", "chapter_id": "uuid", "segment_id": "uuid", "data_base64": "..." }
  ]
}
```

`audios` is used when the frontend has already uploaded segment audio blobs to the backend as a separate step. See §7.

Response:

```json
{
  "results": [
    { "project_id": "uuid-1", "status": "ok", "audio_uploaded": 5, "audio_failed": 0 },
    { "project_id": "uuid-2", "status": "error", "message": "..." }
  ]
}
```

### 5.5 Existing TTS endpoints

Existing `POST /api/tts/synthesize` and `POST /api/mimo-tts/*` continue to work for the legacy TTS flow. When the request body contains `segmented_project_id`, the result is written to the segmented project directory instead of `tts_results`, and `tts_results.source='segmented_tts'` is set for compatibility.

### 5.6 Error codes

| Status | `error` | When |
|---|---|---|
| 400 | `chapter_mismatch` | chapter_id not part of project |
| 404 | `project_not_found` / `segment_not_found` | |
| 409 | `audio_missing` | segment's `current_audio_path` points to a file that no longer exists |
| 422 | `validation_error` | schema mismatch, missing fields, unsupported schema_version |
| 500 | `io_error` / `encode_failed` | filesystem or ffmpeg failure |
| 502 | `tts_engine_error` | upstream TTS call failed |
| 207 | partial migration | mixed result in `/migrate` |

## 6. Project Asset Directory

### Layout

```text
backend/uploads/segmented/{project_id}/
├── manifest.json
├── original.txt
└── chapters/
    └── {chapter_id}/
        ├── original.txt
        ├── segments.json
        └── segments/
            ├── {segment_id}.mp3
            ├── {segment_id}.ssml
            ├── {segment_id}.txt
            └── {segment_id}.prev.mp3   (only when keep_previous=true)
```

### File meanings

| File | Contents | Written on |
|---|---|---|
| `manifest.json` | Snapshot of project + chapters + segments | `PUT`, `synthesize` |
| `original.txt` | Project-level original text | `PUT` |
| `chapters/{cid}/original.txt` | Chapter-level original text | `PUT` |
| `chapters/{cid}/segments/{sid}.mp3` | Segment audio | `synthesize` |
| `chapters/{cid}/segments/{sid}.ssml` | SSML snapshot | `PUT`, `synthesize` |
| `chapters/{cid}/segments/{sid}.txt` | Segment text snapshot | `PUT` |
| `chapters/{cid}/segments.json` | Chapter segments metadata | `PUT` |

### Consistency rules

- DB is written first; files second. A DB transaction failure leaves the filesystem untouched and surfaces as a 5xx to the client.
- `synthesize` writes the mp3 to a temp path, then renames atomically to `{segment_id}.mp3`, then updates the DB in a single transaction.
- `DELETE` removes DB rows first; filesystem deletion is best-effort. Residual directories are logged for a background sweep.
- Old `previous_audio_path` mp3 is removed when a newer audio is generated, unless `keep_previous=true`.

### Optional startup scan

A startup-time directory integrity scan is **out of scope for v1**. Missing audio files are detected lazily: when a `GET audio` is requested and the file is missing, the backend sets `audio_missing = true` on the segment row and the frontend renders the segment as needing regeneration.

## 7. Frontend Storage Adapter and Sync

### IndexedDB draft record (backend mode)

```ts
interface ProjectDraftRecord {
  project_id: string
  draft: SegmentedProject         // unsynced or unconfirmed state
  base_updated_at: string         // backend version this draft was derived from
  updated_at: string              // local last-edit time
  dirty: boolean                  // unsynced changes
  last_save_attempt_at?: string
  last_save_error?: string
}
```

### Sync flow

**Open project**

```text
GET /api/segmented-projects/{id}
  → backend_updated_at
  compare with local draft.base_updated_at
    ├─ no draft → use backend, write clean copy
    └─ draft present
         ├─ draft.base_updated_at < backend_updated_at
         │   → conflict prompt: "Local draft based on older version.
         │                       Restore local edits or use backend?"
         └─ equal → use draft
```

**Edit**

```text
user edit
  ↓
reducer updates state
  ↓
write draft to IndexedDB
  ├─ draft.updated_at = now
  └─ draft.dirty = true
  ↓
schedule debounced PUT (≈1s)
```

**Debounced PUT success**

```text
PUT /api/segmented-projects/{id}
  ↓
project.updated_at returned
  ↓
draft.base_updated_at = new server time
draft.dirty = false
draft.last_save_error = undefined
```

**Debounced PUT failure**

```text
draft.dirty = true
draft.last_save_error = <message>
base_updated_at unchanged
→ next debounce retries, or flushPendingSave on close
```

**Synthesize**

```text
POST /segments/{sid}/synthesize
  ↓
backend updates DB + writes mp3 + returns updated segment
  ↓
frontend reducer applies returned segment
draft.dirty = false (server is now in sync with this update)
```

**Close / unload**

```text
flushPendingSave()
  if dirty and PUT fails → warn user
  best-effort fetch keepalive or sendBeacon as fallback
```

### Triggers summary

| Trigger | React state | IndexedDB draft | Backend |
|---|---|---|---|
| User edit | ✓ | ✓ immediate | debounced PUT |
| Delete segment | ✓ | ✓ immediate | immediate or debounced PUT |
| Apply resegment | ✓ | ✓ immediate | immediate PUT |
| Synthesize segment | ✓ (apply response) | unchanged (clean) | API persists |
| Switch projects | — | dirty=true | flush |
| Reload page | load from backend | use draft (with conflict check) | pull |
| IndexedDB migration | — | migrated → clean | migrate API |

### Invariants

1. The backend is always the source of truth.
2. `base_updated_at` is updated only on a successful `PUT` round-trip.
3. A `dirty=true` draft is recoverable after a reload.

## 8. Frontend Migration Flow

### Trigger

When `useStorageMode.mode === 'backend'` and the user enters the segmented editor for the first time, the frontend checks whether IndexedDB has any project records. If so, it shows a migration prompt.

### Prompt

```text
检测到本地有 N 个分段项目，是否迁移到后端存储？
[迁移]   [保留本地]   [稍后再说]
```

If the user dismisses with "稍后再说", a persistent lightweight banner remains:

```text
⚠ 本地有 N 个项目未迁移  [迁移]
```

### Two-step migration

**Step 1 — Migrate project structures**

```text
POST /api/segmented-projects/migrate
body: { projects: [...] }   // projects' current_audio_id pointing at IndexedDB tts_results
```

The backend creates the DB rows and project directories, and writes text / SSML / manifest files. Audio files are **not** copied in this step (the IndexedDB blobs are not on the wire).

**Step 2 — Upload segment audio blobs**

For each segment with a non-null `current_audio_id`, the frontend reads the corresponding `tts_results` blob from IndexedDB and uploads it:

```text
POST /api/segmented-projects/{project_id}/chapters/{chapter_id}/segments/{segment_id}/audio
body: { data_base64: "..." }
```

The backend writes the file as `chapters/{chapter_id}/segments/{segment_id}.mp3` and updates `current_audio_path`. The frontend tracks per-project progress and shows a progress indicator.

### After migration

- Successful projects: removed from `segmented_projects` and `tts_results` IndexedDB stores
- Failed projects: remain in IndexedDB with a retry option
- If migration is interrupted, the next session detects both backend projects (by `id`) and IndexedDB projects and offers "重新迁移 / 跳过"

### Schema version compatibility

The frontend `migrateV1` (`useSegmentedProject.ts:39`) already converts v1 projects to v2. Only `schema_version=2` projects are sent to the backend.

## 9. Persistence Timestamp Tracking

Both frontend and backend maintain a `updated_at` for every persisted record.

### Backend

- `segmented_projects.updated_at`, `segmented_project_chapters.updated_at`, `segmented_project_segments.updated_at` are touched on any change to the respective row.
- The project-level `updated_at` is **also** touched on:
  - segment text or parameter change
  - segment audio synthesize (because generated fields changed)
  - chapter or project-level metadata change
- The service layer enforces this in a single explicit `touch` per affected row, inside the same transaction as the data write.

### Frontend (IndexedDB draft)

- `draft.updated_at` is set on every local edit
- `draft.base_updated_at` is set to the backend response's `updated_at` after a successful `PUT`
- These two timestamps enable conflict detection on project open and recovery after a crash

### `audio_format`

Stored as the literal string `"mp3"` for any new audio. The TTS engines natively output wav / pcm; the backend transcodes to mp3 before writing to the project directory. If ffmpeg is unavailable at runtime, the segment row is written with `audio_format='wav'` and the wav file is kept under the project directory; the frontend surfaces this as a "needs conversion" notice.

## 10. Error Handling and Edge Cases

### API error → frontend behavior

| Status | Behavior |
|---|---|
| 404 | Reload project from backend; show "project removed" toast |
| 400 / 422 | Show validation error; do not auto-retry |
| 409 audio_missing | Mark segment as `audio_missing`; offer regenerate |
| 5xx | Mark save as failed; retry with exponential backoff (1s, 2s, 4s, 8s, max 4) |
| Network 0 / 502 / 503 / 504 | Same as 5xx |
| 207 (migrate) | Show per-project status; let user retry failed projects |

### Segment audio state machine (frontend-owned)

```text
empty → ready → stale → generating → ready
                                  ↘ failed → generating
ready → audio_missing → (regenerate) → generating → ready
```

Backend does not store this state; the frontend computes it from `current_audio_path` and the diff between `effective params` and `generated_params`.

### Edge cases

- Empty segment text: allowed in DB; `synthesize` returns 422
- Two tabs editing the same project: second tab's `PUT` is gated by `base_updated_at` and prompts a conflict
- Storage mode switch while editing: `flushPendingSave()` first; abort on failure
- `ffmpeg` missing at runtime: project write still succeeds with wav; segment `audio_format` reflects this
- `schema_version != 2` in request: 422
- Project deletion during PUT race: second operation returns 404

## 12. Risks and Known Compromises

1. **Two-step migration is needed** because IndexedDB audio blobs cannot travel in a single migrate request without bloating the body.
2. **Export (wav / srt concatenation) remains frontend-side** in v1; backend `/export` is reserved but unimplemented.
3. **ffmpeg is a runtime dependency**; documented in `docs/ENV.md` and verified at backend startup. Missing ffmpeg causes the project write to fall back to wav files written under the project directory; the segment row is left with `current_audio_path` pointing at the wav and `audio_format = 'wav'` so the user is informed via a "needs conversion" notice in the UI.
4. **No real multi-device collaboration** beyond the `base_updated_at` conflict check. Two devices editing offline and reconnecting will see a conflict prompt.
5. **No PATCH endpoint** in v1; large projects PUT the full state on every save. Acceptable for projects of typical size; revisit if profiling shows otherwise.
6. **Audio file naming** uses `{segment_id}.mp3` plus a `.prev.mp3` for the previous version. Avoids collisions and keeps a rollback path. Multiple historical versions are not retained.
7. **Existing `tts_results.source = 'segmented_tts'`** is set by the legacy synthesize path for compatibility but is **not** the source of truth in backend mode. Frontend must not rely on `tts_results` for project state.
8. **Frontend migration audio upload is a separate `POST audio`** for each segment. For projects with hundreds of segments, the user can see progress; the operation can be cancelled and resumed.
